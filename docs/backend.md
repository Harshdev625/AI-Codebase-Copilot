# Backend Documentation (Complete Reference)

This document explains **what each backend folder does**, **how the backend is managed**, **which schemas exist**, **which endpoints exist**, and **how the runtime flow works**.

---

## 1) What this backend is

The backend is a FastAPI service that powers:
- authentication and authorization,
- project/repository management,
- indexing source code into chunk + vector stores,
- search/chat over indexed repositories,
- admin and tooling APIs.

Primary entry point: `backend/app/main.py`

---

## 2) Backend folder-by-folder guide

Inside `backend/`:

### `app/`
Main application package.

- `main.py`
  - Creates FastAPI app.
  - Mounts all routers under `/v1`.
  - Runs schema/bootstrap initialization on startup.

- `api/`
  - Request dependencies and all API routers.
  - `dependencies.py` handles current-user auth, role checks, repository access checks.
  - `v1/auth` for login/register/me.
  - `v1/repositories` for projects/repos/indexing.
  - `v1/chat` for chat + conversations/messages.
  - `v1/search` for search.
  - `v1/tools` for controlled tool execution.
  - `v1/admin` for admin operations and metrics.

- `core/`
  - Global config (`config.py`) and security primitives (`security.py`).
  - JWT creation/verification and password hashing live here.

- `db/`
  - `database.py`: SQLAlchemy engine/session.
  - `schema.py`: application schema creation + bootstrap admin logic.

- `graph/`
  - Agent workflow graph and state.
  - `workflow.py` wires planner/retrieval/tool/debug/refactor/answer nodes.
  - `nodes/` contains step implementations.

- `llm/`
  - `model_router.py` routes chat + embedding calls to model providers.

- `models/`
  - `api_models.py`: Pydantic request/response schemas.
  - `domain_models.py`: internal domain dataclasses (e.g., code chunks).

- `rag/`
  - Retrieval and embedding logic.
  - `chunking/` for chunk generation.
  - `retrieval/` for dense/lexical/hybrid retrieval.
  - `embeddings/` for embedding provider abstraction + Ollama provider.

- `services/`
  - Business services:
    - `indexing_service.py`
    - `query_service.py`
    - `qdrant_service.py`
    - `cache_service.py`

- `tools/`
  - Internal tool wrappers and safety checks (`read_file`, `git_status`, command execution policy).

### `tests/`
- `unit/`: isolated tests with mocks/fakes.
- `integration/`: live API/server-style tests.

### `scripts/`
- DB utility scripts (`clear_db.py`, `init_pgvector.sql`).

### `run.py`
- Dev run launcher.

### `pyproject.toml`
- Dependencies, test/coverage config, build config.

### `ai_codebase_copilot_backend.egg-info/` (why this exists)
This folder is **Python package metadata generated by setuptools** (usually after `pip install -e .` or building/installing the package). It contains:
- package name/version metadata,
- dependency metadata,
- source file manifest.

It is needed by Python packaging tooling for editable installs and package resolution. It is not business logic code.

## 3) How backend management works (runtime + data flow)

### Request flow
1. Request hits FastAPI router under `/v1`.
2. Dependency layer validates JWT + user role/membership.
3. Router delegates to services.
4. Services call DB and/or external systems (Ollama, Qdrant, Redis).
5. Response model is validated by Pydantic and returned.

### Indexing flow
1. `POST /v1/index` validates repository access.
2. Creates snapshot/job records.
3. Runs indexing as background task.
4. Files are discovered (git-aware + ignore rules).
5. Chunks are produced and stored in Postgres.
6. Embeddings/Qdrant writes are best-effort.
7. Progress and final status are written to `indexing_jobs` and `repository_snapshots`.

### Chat/search flow
1. Repository access is validated.
2. Retrieval runs (hybrid dense + lexical).
3. Graph workflow processes intent/state.
4. LLM answer generation runs with assembled context.
5. Conversation/message records + run metadata are persisted.

## 7) External dependencies used at runtime

- **Postgres + pgvector**: persistent relational + vector-aware storage.
- **Qdrant**: optional vector index/search backend.
- **Ollama**: local inference for chat/embeddings.
- **Redis**: optional cache layer.

The system is designed to degrade gracefully where possible (e.g., chunk persistence can continue even if vector/embedding services are unavailable).
